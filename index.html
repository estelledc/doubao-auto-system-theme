01

Problem

真正的问题不是背景颜色，而是 SPA 里的主题状态会漂移。

页面启动、路由变化、异步渲染和自身脚本都可能重新写入主题属性。即使主背景变暗，侧栏标题、Markdown 正文、代码块顶栏或弹窗仍可能保留浅色表面与低对比文字。

State drift

页面重写 data-theme 后，局部组件回到旧状态。

Surface leaks

“大面积已变暗”仍可能藏着白块、黑字和不可读 token。

Blanket overrides

全局改写标签或使用 invert() 会误伤图片、品牌色与控件。

02

System

从系统信号到可读界面的四层适配链。

  1. Sense
    监听系统主题

    matchMedia('(prefers-color-scheme: dark)') 提供目标状态，并响应系统变化。

  2. Synchronize
    同时驱动 html 与 body

    同步 4 个主题属性、class 与 color-scheme，减少组件各读一套状态。

  3. Repair
    只修补已知表面

    用有边界的选择器处理侧栏、Markdown、代码、输入区和浮层，不做全局反色。

  4. Audit
    把“看起来正常”变成结构化输出

    __doubaoThemeAudit() 返回主题属性、对比度、暗字岛、代码块和亮色残留。

06 / Limitations

适配器依赖外部页面结构，失效是需要被设计的状态。

这是个人维护的非官方用户脚本，只覆盖 Chrome + Tampermonkey 与 doubao.com/chat/*。豆包改版后，选择器与主题变量可能需要更新；页面没有真实安装量、长期稳定性或完整浏览器矩阵证据，也不承诺永久兼容。

脚本不读取、保存或上传聊天内容，元数据声明 @grant none；但安装任何用户脚本前，仍应先阅读源码并确认权限范围。
